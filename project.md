# PRD / ТЗ: Лендинг системы геймификации

## 0) Как читать этот файл (чтобы ИИ не путался)

1. Секции `1-7` (Контекст -> Acceptance criteria) считаются “нормализованной” спецификацией.
2. `Приложение A` содержит сырой текст из исходников (как извлеклось) и может включать странные переносы строк (особенно из PDF).

## 1) Контекст

Нужно разработать современный лендинг системы геймификации. В основе платформы механика челленджей уровней `LIGHT / MEDIUM / HARD`. Пользователи выполняют задания и получают награды.

## 2) Цели

1. Показать механику геймификации и мотивацию (награды).
2. Дать пользователю понятный путь от ознакомления до входа в систему.
3. Сохранить единый стиль по брендбуку “ФР”.

## 3) Роли и доступы (минимум из документов)

### Роли

- `participant` (участник): выполняет задания/дейлики, получает баллы, видит прогресс, рейтинг, награды/достижения.
- `admin` (админ): отдельный аккаунт, отдельный функционал.
- `pm` (проектный менеджер команды): отдельный аккаунт, отдельный функционал.

Примечание: в одном из документов явно перечислены аккаунты `admin` и `pm`. Роль `participant` подразумевается текстом (“Пользователи выполняют задания…”).

### Авторизация

- вход по логину и паролю
- разный функционал для `admin` и `pm`

## 4) Обязательный контент лендинга (IA)

Лендинг должен содержать следующие секции:

1. `Hero` (главный экран)
2. `Mechanics` (описание механики)
3. `Difficulty` (описание уровней сложности `LIGHT/MEDIUM/HARD`)
4. `AchievementsAndRewards` (блок достижений и наград)
5. `Progress` (демонстрация прогресса, прогресс-бар)
6. `Rating` (автоматизированный рейтинг участников)
7. `AuthCTA` (вход / призыв к авторизации)

## 5) Механика геймификации (функциональные требования)

### 5.1 Уровни сложности

- есть уровни: `LIGHT`, `MEDIUM`, `HARD`

### 5.2 Прогресс

- отображается прогресс выполнения (в т.ч. прогресс-бар на лендинге)

### 5.3 Награды

Таблица из материалов:

| Условие | Награда |
|---|---|
| `LIGHT` | стикерпак |
| `LIGHT + MEDIUM` | брелок |
| `LIGHT + MEDIUM + HARD` | встреча с экспертом |

### 5.4 Дейлики и баллы

- участие в дейликах (как обязательное условие)
- баллы за дейли (как обязательный элемент системы)

### 5.5 Достижения

- должны быть достижения (как обязательный элемент системы)

### 5.6 Ограничения/условия (как в материалах)

- челленджи рассчитаны на месяц с возможностью повторения
- ведение задач в Jira
- регулярные встречи с заказчиком

### 5.7 Примеры челленджей (для контента/карточек)

- “Помог другому члену команды закрыть задачу”
- “Разработал план оптимизации продукта”
- “Предложил и реализовал фичу, которая влияет на оптимизацию бизнес процесса”
- “Предложить и реализовать фичу, которая влияет на бизнес-метрики продукта”

### 5.8 Опционально (можно реализовать)

- ежедневные задания
- магазин наград
- кастомизацию профиля

## 6) Требования к дизайну (из брендбука ФР)

### 6.1 Идея/тональность

- IT, технологичность, инновации, свежесть, гибкость, смелость в решениях, новый взгляд

### 6.2 Цвета (hex)

- `#FFFFFF`
- `#D9D9D9`
- `#111111`
- `#B559F3`
- `#18EFF2`
- `#52F2C5`

### 6.3 Шрифт

- `SB Sans Display Semibold`

### 6.4 Иллюстрации

- пиксельные 3D (неформальные)
- гладкие 3D (официальные)

## 7) Acceptance criteria (AI-friendly чеклист)

### Лендинг

- [ ] На лендинге есть секции: Hero, Mechanics, Difficulty, AchievementsAndRewards, Progress, Rating, AuthCTA.
- [ ] В секции Difficulty перечислены уровни `LIGHT/MEDIUM/HARD` и их смысл.
- [ ] В секции AchievementsAndRewards есть награды и/или достижения.
- [ ] В секции Progress отображается прогресс-бар (демо достаточно, если нет данных).
- [ ] В секции Rating отображается рейтинг участников (демо достаточно, если нет данных).

### Авторизация

- [ ] Есть форма входа по логину/паролю.
- [ ] Можно войти под ролью `admin`.
- [ ] Можно войти под ролью `pm`.
- [ ] После входа роль влияет на доступный функционал (минимум: разные разделы/экраны или разные действия).

### Стиль

- [ ] Используется палитра из брендбука.
- [ ] Используется шрифт из брендбука (или максимально близкая замена, если недоступен).
- [ ] Визуальная концепция выдержана в “технологичность/свежесть/смелость”.

## 8) Открытые вопросы (чтобы не гадать)

1. Что именно должен уметь `admin` (CRUD челленджей/награды/пользователи/рейтинг)?
2. Что именно должен уметь `pm` (управление своей командой/подтверждение выполнений/просмотр прогресса)?
3. Откуда берутся данные для “ведение задач в Jira” и “встречи с заказчиком”: это ручные отметки или интеграция с Jira?
4. Как считать рейтинг: по баллам, по закрытым челленджам, по комбинации?
